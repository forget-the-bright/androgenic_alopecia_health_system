## 1. 数据库设计概述
### 1.1 设计原则
- 遵循第三范式（3NF），减少数据冗余；
- 核心业务表增加主键、索引，提升查询性能；
- 敏感数据（如密码）加密存储，保障数据安全；
- 字段设计兼顾扩展性，预留必要的冗余字段；
- 所有表增加通用审计字段（创建时间、更新时间），便于追溯。

### 1.2 命名规范
- 表名：小写+下划线，前缀区分业务（sys_：系统表，user_：用户业务表，ai_：AI分析表）；
- 字段名：小写+下划线，使用语义化命名（如user_id、upload_time）；
- 主键：统一命名为id，类型为BIGINT自增；
- 外键：关联表主键+_id（如user_id关联sys_user表的id）。

## 3. 数据库初始化建表语句
```sql
-- ########################### 系统用户表 ###########################
CREATE TABLE `sys_user` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '用户唯一标识',
  `username` VARCHAR(50) NOT NULL COMMENT '登录账号（唯一）',
  `password` VARCHAR(100) NOT NULL COMMENT '加密密码（MD5/SHA256）',
  `real_name` VARCHAR(20) DEFAULT NULL COMMENT '真实姓名',
  `phone` VARCHAR(11) DEFAULT NULL COMMENT '手机号码',
  `email` VARCHAR(50) DEFAULT NULL COMMENT '电子邮箱',
  `role` TINYINT NOT NULL DEFAULT 0 COMMENT '角色（0：普通用户，1：系统管理员）',
  `language` VARCHAR(10) NOT NULL DEFAULT 'zh-CN' COMMENT '语言偏好（zh-CN：中文，en-US：英文）',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '账号状态（0：禁用，1：启用）',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `last_login_time` DATETIME DEFAULT NULL COMMENT '最后登录时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_username` (`username`),
  KEY `idx_role` (`role`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统用户表';

-- ########################### 毛发照片表 ###########################
CREATE TABLE `user_hair_image` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '照片唯一标识',
  `user_id` BIGINT NOT NULL COMMENT '关联用户ID',
  `part` VARCHAR(20) NOT NULL COMMENT '拍摄部位（头顶/前额/两侧等）',
  `file_url` VARCHAR(255) NOT NULL COMMENT 'MinIO文件访问URL',
  `minio_path` VARCHAR(255) NOT NULL COMMENT 'MinIO文件存储路径',
  `file_name` VARCHAR(100) NOT NULL COMMENT '文件名（含后缀）',
  `file_size` BIGINT NOT NULL COMMENT '文件大小（字节）',
  `file_type` VARCHAR(10) NOT NULL COMMENT '文件类型（jpg/png）',
  `upload_date` DATE NOT NULL DEFAULT CURRENT_DATE COMMENT '上传日期',
  `upload_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '上传时间',

  `analysis_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '分析完成时间',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  KEY `idx_user_analysis_time` (`user_id`,`analysis_time`),
  KEY `idx_image1` (`image_id1`),
  KEY `idx_image2` (`image_id2`)

-- ########################### 用药方案表 ###########################
CREATE TABLE `user_medicine` (
```

## 4. 数据安全与约束
### 4.1 外键约束（可选）
如需启用外键约束，可添加以下语句（根据实际场景选择）：
```sql
-- 用户照片表关联用户表
ALTER TABLE `user_hair_image` ADD CONSTRAINT `fk_hair_image_user` FOREIGN KEY (`user_id`) REFERENCES `sys_user` (`id`) ON DELETE CASCADE;
```

### 4.2 数据加密建议
- 密码字段：使用SHA256+盐值加密，避免明文/简单MD5；
- 敏感信息（手机号、邮箱）：可采用脱敏存储（如138****1234）；
- MinIO文件URL：生成带时效的临时签名URL，避免永久访问。

### 4.3 备份策略
- 全量备份：每日凌晨执行一次全库备份；
- 增量备份：每小时执行一次增量日志备份；
- 备份文件保留周期：至少30天，异地存储。

## 5. 扩展说明
- 表结构可根据实际业务需求扩展字段（如用户表增加“年龄”“脱发等级”等）；
- 索引可根据实际查询场景优化，避免过度索引；
- 对于大数据量表（如user_hair_image、sys_operation_log），建议分表/分区存储（按用户ID/时间）；
- 所有时间字段建议统一使用UTC+8时区，避免时区错乱。
